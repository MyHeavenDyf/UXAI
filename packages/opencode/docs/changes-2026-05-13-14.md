### Commit 4: `b29bf18eb` (5/14) — 修复 make 页面 HTML 卡片检测：从 write tool 的 state.input 提取 HTML 内容

**问题**: `octo_make` agent 使用 `write` tool 写入 HTML 文件，但之前卡片检测逻辑只检查 `state.attachments` 和 `state.output`，没有检查 `state.input`（HTML 内容实际存储的位置）。

**根因分析**:
- `state.attachments` = `false`（write tool 不生成附件）
- `state.output` = `"Wrote file successfully."`（只是成功消息）
- `state.input` = 实际 HTML 内容（之前未检测）

**修改内容**:

| 文件 | 改动 |
|------|------|
| `make/components/insight-turn.tsx` | 重写 `outputCard` memo，新增 `state.input` 检测 |
| `make/components/result-viewer/index.tsx` | 改用 `<Switch>/<Match>` 模式（与 insight 页面一致） |

**检测逻辑优化**:

```
对每个 tool part，按优先级检查：
1. state.attachments — HTML 附件（mime text/html）
2. state.input — write tool 的输入（检查 input.content/text/data）
3. state.output — 工具输出文本
最后兜底：text parts
```

## 三、关键技术决策

### 1. octo.tsx vs app.tsx 双入口同步

桌面端使用 `octo.tsx` 作为入口（不是 `app.tsx`）。所有路由、lazy import、isOctoPage 判断必须同时在两个文件中更新。5/14 的技能库页面首次提交时遗漏了 `octo.tsx`，导致桌面端路由报错回退到 chat 页面。

**教训**: 修改路由/布局相关代码时，必须同时检查 `app.tsx` 和 `octo.tsx`。

### 2. write tool 的数据结构

`write` tool 的 `state` 对象结构：
- `input` — 工具输入参数（包含 `path`、`content` 等）
- `output` — 工具执行结果（通常是简短消息）
- `attachments` — 文件附件（write tool 通常不生成）

HTML 内容在 `state.input` 中，不在 `state.output` 或 `state.attachments` 中。卡片检测必须检查 `input` 字段。

### 3. SolidJS 响应式模式

Insight 页面使用 `<Switch>/<Match>` 配合 Show 回调中的 `tab()` getter 实现响应式类型切换。Make 页面的 ResultViewer 应遵循相同模式，避免使用非响应的 `const t = tab()` 捕获。
